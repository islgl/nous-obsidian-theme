## [Unreleased]

### Changed
- **Handwriting mode: body text scales up** ‚Äî reading-view bumped from 1.25em to 1.45em; edit-mode from 1.05em to 1.2em. Italianno's small x-height and Xingkai SC's dense strokes both benefit from the extra room; headings (which are measured in em relative to body) scale up automatically.
- **Handwriting mode: body font swap + dedicated table style** ‚Äî body English font moves from **Petit Formal Script** (17th-c. secretary hand ‚Äî rounded and uniform) to **Italianno** (19th-c. English copperplate ‚Äî thin, pointed, high-contrast strokes with long ascender loops). Same warm cursive family on paper, but with ornament and character instead of round uniform blobs. Reading-view font-size bumps from 1.08em to 1.25em to compensate for Italianno's smaller x-height; edit-mode size from 1em to 1.05em. Pinyon Script added as a fallback. Also: tables get a completely new handwriting-mode treatment ‚Äî dashed warm rules (top / bottom / header separator), no verticals, Italianno script headers at 1.55em in terracotta, quiet serif body cells at 0.95em so names and numbers stay readable. Feels like a hand-ruled notebook instead of the booktabs LaTeX table the base theme uses.
- **CHANGELOG push hook** ‚Äî relaxed from "CHANGELOG must be staged or in unpushed commits" to "CHANGELOG must differ from origin/main in any way (working tree, staged, or committed)." Fixes false negatives when Claude chains `git add && git commit && git push` in one shell call ‚Äî the hook now fires on the intent of the rule, not on mid-pipeline filesystem state.

### Fixed
- **Mermaid label halos: thin strokes, round caps** ‚Äî the 4px halo on edge/message/relationship/pie-title text (and 2px text-shadow on HTML labels) was producing visible black-dot artifacts on serif terminals, "i" dots, and sharp glyph corners ‚Äî exactly the fuzz the halo was supposed to fix. Narrowed every SVG stroke to 2.5px (2px on the smaller legend text) and added `stroke-linecap: round` alongside the existing `stroke-linejoin: round`. HTML foreignObject text-shadow offsets dropped from 2px to 1.5px in both modes. Halo is still wide enough to clear a 1.4px connector, but no longer thick enough to blob around dots or terminals. (One `relationshipLabel` rule was missed in the first pass because of a font-size mismatch in the find-and-replace; picked up in a follow-up.)
- **Mermaid pie title + legend legibility** ‚Äî title "Hours per week" and legend entries were rendering black via Mermaid's SVG-embedded `#svgid .pieTitleText { fill: black }` style; on the dark body this was invisible and even on light parchment it didn't read as part of the Nous text system. Rules now target both `.pieTitleText` AND `text.pieTitleText` (also `g.legend text` for extra specificity) with a paint-order-stroke halo in the page background color, and a dark-mode override keyed off `body.theme-dark` (one notch higher specificity than plain `.theme-dark`). Belt + suspenders: even if upstream CSS repaints the fill black, the halo preserves a ring of background around each glyph so the title and legend always read.
- **Mermaid sequence message text legibility** ‚Äî message labels (`open`, `present signs`, `interpret`, etc.) sitting near the connector lines were reported unreadable in dark mode. The dark-mode color rule was already in place, but without a halo the glyphs could not survive overlapping the message stroke. Added the same paint-order-stroke halo treatment to `.messageText` / `text.messageText` in BOTH modes (page-bg color, 4px stroke, round joins) so message text floats clear over the stroke regardless of light/dark theme. Also anchored the light-mode rule with an extra `text.messageText` selector so element-type-qualified versions don't escape.
- **Mermaid ER in dark mode, actually applied** ‚Äî the previous "every box is a cream card" commit did NOT reach ER entities in practice because three older `.theme-dark .markdown-preview-view .mermaid .entityBox` rules (left over from a prior attempt) carried specificity (0,4,0) and outranked the consolidated cream-card block at specificity (0,3,0). Both had `!important`, so the higher-specificity old rule won the cascade and kept painting entities `--nous-warm-sand` dark. Removed the three stale rules (entityBox dark fill, attributeBox dark stroke, relationshipLine dark stroke) ‚Äî the consolidated block now wins as intended, and ER entities render on the cream card like every other diagram type.
- **Mermaid dark mode: every node box is a warm cream card** ‚Äî extended the cream-card decision from sequence actors to EVERY diagram type (flowchart nodes, state-diagram states, class-diagram boxes, ER entities, sequence actors/notes/labelBoxes). Every diagram embeds its own inline `<style>` scoped by the SVG id ‚Äî and each flavor (classDiagram, stateDiagram, ER) forces different text fills (`#333`, `#131300`, `#9370DB`) on tspans/labels with a specificity we can only beat via ID selectors we don't have access to. One cream card surface (`#e8e2d2`) + one warm near-black text override (`#2d2a24`) + one tan border (`#a67b5b`) ends the cascade war for all four diagram types at once. Text inside boxes now reads consistently in dark mode regardless of which Mermaid diagram flavor is drawing it. ER attribute rows alternate between the main cream and a one-notch-darker `#ddd7c7` so the striping reads on the card rather than as transparent dark strips punched through it.
- **Mermaid sequence diagrams in dark mode** ‚Äî actor / labelBox / note boxes were being painted dark (`#30302e`) but Mermaid's SVG-embedded `<style>` forces `text.actor > tspan { fill: black }` with a specificity we can't beat without unhinged selectors. Result: black text on dark box = invisible. Fix inverts the strategy: keep those three element types as **warm cream cards** (`#e8e2d2`) in dark mode too ‚Äî Mermaid's forced-black tspan text then reads fine against a light card. This decision was later generalized to every diagram type (see above entry).
- **Mermaid dark-mode: fresh cohesive theme** ‚Äî scrapped the piecemeal `.theme-dark` overrides scattered through the flowchart / ER / pie / sequence blocks and replaced them with a single dark theme definition keyed off `--mm-*` custom properties (text / surface / stroke / edge / hairline / halo). Dark text is now `#d5d2c6` ‚Äî a warm off-white, not pure white ‚Äî so diagrams read without glare. Edges use a warm medium gray `#7a7770`; arrowheads lift one notch to `#9d9a92`; node strokes use a muted terracotta `#b25a3a` instead of the bright dark-mode terracotta. Node surfaces stay at `--nous-warm-sand` for card-feel; cluster surfaces sit one step below at `--nous-ivory` so nesting reads. Pie slice palette override (coral / golden tan / taupe / apricot / antique gold / warm khaki / burnt sienna) already landed previously and feeds the new theme unchanged.
- **Mermaid fallback selectors ‚Äî attribute-value safety net** ‚Äî Mermaid's class naming varies across versions and its output in Obsidian doesn't always expose the classes our CSS guessed (ER entity boxes and markers were a case in point ‚Äî the class-based rules missed, so the diagram kept Mermaid's cool default `#ECECFF` lavender). Added a third line of attack: `.mermaid rect[fill="#ECECFF"]`, `[stroke="#9370DB"]`, `[fill="#000"]:not(text)` ‚Äî catch Mermaid's default palette by *value* and repaint to the Nous warm palette in both modes. Scoped to `.mermaid` so external SVGs on the page aren't touched. Where class selectors hit, class rules win on specificity; where they miss, attribute rules cover.
- **Mermaid dark-mode legibility** ‚Äî nodes now lift to `--nous-warm-sand` in dark mode instead of `--background-primary-alt`, so node cards read as cards rather than as dark holes punched into an already-dark page. Cluster rects, sequence notes, and actor surfaces get the same treatment. Gantt grid lines switch to `--nous-ring-warm` in dark mode too.
- **Mermaid edge-label legibility** ‚Äî edge labels (`yes` / `no` branch guards, sequence-diagram messages, ER relationship verbs) had become unreadable after the background rect was stripped ‚Äî muted italic text directly on top of the connector stroke. Replaced with a typography-halo: SVG `<text>` uses `paint-order: stroke fill` with a 4px stroke in the page background color (the stroke is painted *under* the glyph fill so it reads as a halo, not an outline); HTML labels rendered via `foreignObject` get an 8-direction `text-shadow` at 2px in the same color. Result: no visible box, text still floats clear over the stroke. Label size bumped to 1.05em / weight 500 ‚Äî readable at a glance, not a whispered caption.
- **Mermaid ER harmony (actually applied this time)** ‚Äî the previous attempt used selectors like `.mermaid .er .entityBox` (descendant combinator), assuming Mermaid wrapped the diagram in a `<g class="er">` parent. It doesn't: Mermaid writes `class="er entityBox"` as two classes on the *same* element, so the descendant selector matched nothing and the rules were silently dropped. Fixed by targeting each class directly (`.mermaid .entityBox`, `.mermaid .relationshipLine`, etc.) and bumping specificity with `.markdown-preview-view` to outrank Mermaid's inline `<style>`. ER cardinality markers (one / many / zero-or-one) in `<defs><marker>` now also get the warm charcoal fill instead of defaulting to black.
- **Mermaid pie harmony (actually applied this time)** ‚Äî same class-shape mistake: slices in v10+ are all `<path class="pieCircle">` with colors applied via a `fill="‚Ä¶"` SVG attribute, not via `.pieN` classes. Recolor now works by targeting `.markdown-preview-view .mermaid path.pieCircle:nth-of-type(7n+k)` (CSS `!important` beats a presentation attribute).
- **Mermaid pie legend** ‚Äî the first attempt assumed Mermaid emitted `<rect class="legend0">` on the swatches; the rects actually have *no class* ‚Äî color lives in an inline `style="fill: rgb(‚Ä¶); stroke: rgb(‚Ä¶)"` on the rect, with `.legend` sitting on the parent `<g>`. New selector is `.mermaid g.legend:nth-of-type(7n+k) > rect` and overrides both `fill` and `stroke` (Mermaid sets both inline). The decorative `circle.pieOuterCircle` border also gets a transparent fill + warm hairline so it stops fighting the slice palette.

## [1.0.0-alpha.1] ‚Äî 2026-04-18

First public alpha.

### Added
- **Core theme** (`theme.css`) ‚Äî warm parchment palette (`#f5f4ed` canvas, `#c96442` terracotta accent), near-black `#141413` dark variant, serif body in both edit and reading view.
- **LaTeX booktabs tables** ‚Äî 2px top/bottom rules, 1px mid rule, no verticals, small-caps header row, tabular lining figures.
- **Theorem-style callouts** ‚Äî 13+ built-in Obsidian callouts restyled, plus custom `[!theorem]` / `[!lemma]` / `[!proposition]` / `[!corollary]` / `[!proof]` / `[!definition]` / `[!remark]` / `[!example]`; `[!proof]` auto-appends ‚àé.
- **Mermaid recoloring** ‚Äî flowchart / sequence / state / class / gantt / pie / ER diagrams mapped to the warm palette.
- **Ring-shadow depth system** ‚Äî inset / 1px rings in warm grays in place of drop shadows.
- **Content typography** ‚Äî small-caps at H5/H6, italic at H4, tight display letter-spacing on the inline title, old-style figures in body text.
- **Style Settings integration** ‚Äî fonts, body size, line height, reading column width, justify toggle, numbered-headings toggle, drop-cap toggle, H5/H6 small-caps toggle.
- **Optional snippets** (in `snippets/`):
  - `nous-paper-mode.css` ‚Äî LaTeX article jacket, triggered by `cssclasses: [paper]` (¬ß-numbered headings, drop cap, justified body, figure/table captions, ¬ß hr ornament).
  - `nous-handwriting.css` ‚Äî quill-and-parchment jacket, triggered by `cssclasses: [handwriting]` (Petit Formal Script + Xingkai SC Ë°åÊ•∑, flower ornaments ‚ùÉ ‚úø ‚ùÅ on H1/H2/H3, centered pull-quote blockquotes).
  - `nous-theorem-counters.css` ‚Äî amsthm-style auto-numbering for `theorem` / `lemma` / `proposition` / `corollary`.
  - `nous-classical-figures.css` ‚Äî OpenType old-style figures in body, tabular lining figures in tables.
- **Mixed-language typography** ‚Äî CJK `size-adjust` so Chinese lines align to Latin x-height in mixed headings.
- **Brand assets** ‚Äî mark, inverse mark, lockup, OG image (`assets/logo/`).

### Known limitations
- Third-party plugin UIs (Dataview, Kanban, Advanced Tables, etc.) are not yet fully covered.
- Variable names may be renamed before 1.0.0.
- Not yet submitted to the Obsidian community theme catalog ‚Äî install manually from the Releases page.
